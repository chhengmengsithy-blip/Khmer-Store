# V3 marketplace rebuild: simplified schema, full auth, admin panel

The entire application has been rebuilt from a 25-table e-commerce schema (with escrow, wallets, fraud monitoring, and complex role hierarchies) down to a 6-table classified-ads marketplace. The new system implements Supabase email+password and Google OAuth authentication, a user-facing marketplace with search/filter/sort/favorites/messaging, and a complete admin panel for managing users, listings, categories, and reports. The V2 service layer stubs (escrow, payments, fraud, notifications) have been removed entirely, and the database types, constants, and frontend types now reflect the simplified model.

Watch for: search queries interpolate user input into `ilike`/`or` filter strings without sanitization, creating a PostgREST filter injection vector **(confirmed)**; `increment_views` calls an RPC function not defined in the migration -- view counts never increment **(confirmed)**; the `AuthProvider` hardcodes `role: "user"` on every session, never fetching the actual profile role, so client-side admin UI gating won't work **(confirmed)**; no file-type validation on image uploads beyond extension **(confirmed)**.

## High-level view

The schema consolidation is clean: six tables (profiles, categories, listings, favorites, messages, reports) with RLS policies enforced at the database layer, a trigger that auto-creates a profile on signup, and seed data for categories.

Authentication flows through Supabase Auth with server actions for email sign-in/sign-up and a client-side `signInWithGoogle` function. The `AuthProvider` component initializes a Zustand store from the session, but it always sets `role: "user"` regardless of the actual profile role, which means the header and any client-side role gates never know if the current user is an admin.

The admin panel has a layout-level auth gate (checking `profiles.role`) and a `verifyAdmin()` helper used by every server action. Admin authorization happens at the server-action level, not middleware, so there's no way to bypass it via direct action invocation. The gap is client-side: the header will never render an "Admin" link because the Zustand store always says `role: "user"`.

The marketplace page moved from a client-side `useSearchParams` approach to an async server component that fetches listings with URL-driven search, category, and sort params. The search filter constructs Supabase `or()/ilike()` queries by string-interpolating user input, which creates a filter injection risk.

Messaging is a client-side page that loads conversations via server actions and groups them by the other participant. There is no real-time subscription, so incoming messages from the other party won't appear without re-navigation.

## Search filter injection in `getListings`

The marketplace search constructs a Supabase filter via string interpolation:

```typescript
query = query.or(
  `title.ilike.%${params.search}%,description.ilike.%${params.search}%`
);
```

Supabase's PostgREST `or()` syntax parses the filter string. A search term containing `,` or `(` or `)` characters can break out of the intended `ilike` filter and inject additional filter clauses. For example, a search string like `%,id.eq.some-uuid` could cause PostgREST to evaluate an unintended filter condition. The risk is information disclosure (bypassing the `status = active` filter) rather than data modification, since this is a SELECT query. The fix is to sanitize `params.search` by escaping PostgREST-special characters or switching to individual `.ilike()` method calls which handle escaping internally.

## Missing `increment_views` database function

```typescript
await supabase.rpc("increment_views", { listing_id: id });
```

The migration `001_schema.sql` does not define an `increment_views` function. The old V2 migration had `increment_product_views` but it was deleted. Every listing detail page load triggers this call, and Supabase returns a `42883` function-not-found error that gets silently swallowed (no error handling on the return value). View counts will never increment. The fix:

```sql
CREATE OR REPLACE FUNCTION increment_views(listing_id UUID)
RETURNS VOID AS $$
BEGIN
  UPDATE listings SET views_count = views_count + 1 WHERE id = listing_id;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

## AuthProvider role hardcoding

In `auth-provider.tsx`, the user object stored in Zustand always has `role: "user"`:

```typescript
setUser({
  id: session.user.id,
  email: session.user.email || "",
  role: "user",  // <-- never fetched from profiles
  created_at: session.user.created_at,
  updated_at: session.user.updated_at || session.user.created_at,
});
```

The header component reads from `useAuthStore()` but never shows an admin link because the role is always "user" on the client. Admin access currently works only through direct URL navigation to `/admin`, where the server-side layout gate checks the real role. The fix: after getting the session, query the profiles table and use `profile.role`.

## Admin self-demotion and mutual demotion

`updateUserRole` does not check whether `userId === currentUser.id`. An admin can accidentally demote themselves and permanently lock themselves out of the admin panel (since no other mechanism can restore the role without direct DB access). There's also no guard preventing one admin from stripping another admin's role. For a multi-admin system, at minimum prevent self-demotion.

## Reports visibility for reporters

RLS policies on `reports`:
- `reports_insert_own`: authenticated users can insert where `reporter_id = auth.uid()`
- `reports_select_admin`: only admins can SELECT

A user who files a report has no SELECT access to their own report afterward. They cannot check whether it was resolved or dismissed. If the product intends to show report status to reporters, add:
```sql
CREATE POLICY "reports_select_own" ON reports
  FOR SELECT TO authenticated
  USING (reporter_id = auth.uid());
```

## Messaging without real-time

The messages page fetches conversations on mount and thread messages on selection via server actions. There's no Supabase Realtime subscription. The `sendMessage` action refreshes the local thread after sending, but incoming messages from the other party won't appear until the user navigates away and back. For a launched product, polling or a Realtime channel subscription would be needed.

## Image upload without type validation

Both `createListing` and the settings page upload files to Supabase storage with path patterns like `${user.id}/${Date.now()}-${random}.${ext}`. The only "validation" is the file extension derived from the original filename. A user could upload an HTML file renamed to `.jpg`, which would be served with an image content type from Supabase's CDN (Supabase infers content type from extension). For a public bucket, this could serve arbitrary HTML. Validate the actual file bytes (magic number check) or restrict uploads via Supabase storage policies to image MIME types only.
