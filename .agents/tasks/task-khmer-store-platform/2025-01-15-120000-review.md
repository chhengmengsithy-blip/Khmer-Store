# Ultra-premium luxury 3D marketplace platform (greenfield)

A full-stack marketplace platform built from scratch with Next.js 15, Supabase, and a dark-luxury design system. The architecture spans 23 pages, 23+ database tables with RLS policies, a multi-provider payment/escrow service layer, anti-fraud systems, and a 3D hero section. The project builds and lints cleanly.

Watch for: Webhook signature verification is stubbed and will accept any request in production (confirmed). The middleware role check relies on a client-settable cookie with no server-side enforcement at the route handler level (confirmed). The rate limiter is in-memory and resets on every deployment/cold-start, providing no real protection in a serverless environment (confirmed). The verification flow's file upload on the client side never actually calls the server action to persist data (confirmed).

## High-level view

The database schema covers 23+ tables with foreign keys, check constraints, and comprehensive RLS. The `process_escrow_release` function correctly deducts commission before crediting the seller wallet but has no concurrency guard against double-release from parallel invocations.

The middleware uses a `user-role` cookie as the authority for admin/seller gating. If the cookie is absent, the check is skipped entirely — and most admin pages are client components rendering mock data without any server-side authorization.

The service layer (payment, escrow, fraud, notification) defines clean typed interfaces but every implementation returns fabricated data. The webhook signature verification accepts all requests with correctly-formatted headers regardless of cryptographic validity.

Interactive frontend flows (checkout, KYC upload, product creation) manage state locally but never submit to the server. The `handleSubmit` in `verification-flow.tsx` sleeps for 1.5 seconds and returns without calling the server action defined in `verification-actions.ts`.

1. **Webhook signature verification is a no-op** — Both Stripe and PayPal webhook routes check header format but never compute or compare HMAC signatures. Any request with well-formed headers passes. Integrate Stripe SDK `constructEvent` and PayPal certificate verification before deploying.
2. **Admin route gating uses a client-settable cookie** — Middleware reads `user-role` from a cookie the client can freely set. If absent, the check is skipped entirely. Admin pages must perform a server-side role lookup (e.g., via `getUser()` + DB query) before rendering or returning data.
3. **Rate limiter is in-memory and useless in serverless** — The token-bucket implementation uses a per-process `Map`. In Vercel's serverless model, each invocation may be a fresh instance. Replace with Redis or Upstash rate limiting.
4. **KYC verification flow never calls server action** — The `handleSubmit` in `verification-flow.tsx` uses `setTimeout` to fake submission. Wire it to `submitVerification` and `uploadDocument` from `verification-actions.ts`.
5. **Escrow release lacks idempotency guard** — `process_escrow_release` has no `SELECT ... FOR UPDATE` or idempotency key. Concurrent calls could double-credit the seller wallet.
6. **RLS helper `get_user_extended_id()` may return NULL silently** — If a user's `users_extended` row doesn't exist yet (e.g., immediately after sign-up before a trigger creates it), all RLS policies using this function will deny access with no helpful error.
7. **No test coverage** — Zero test files exist. The services, security utilities, and auth actions are all untested.

## Webhook security posture

Both webhook routes (`/api/webhooks/stripe/route.ts`, `/api/webhooks/paypal/route.ts`) have verification functions that always return `true`.

In the Stripe handler, `verifyStripeSignature` checks that the header contains `t=` and `v1=` components and that the timestamp is within 5 minutes, then returns `true` without computing the HMAC-SHA256. Any attacker who sends a correctly-formatted `stripe-signature` header can trigger arbitrary order status changes once the handler bodies are implemented.

The PayPal handler checks for the presence of four headers and immediately returns `true`:

```typescript
if (!transmissionId || !transmissionTime || !certUrl || !transmissionSig) {
  return false;
}
// ...
return true;
```

The `void` statements suppressing unused-variable warnings make it explicit that cryptographic verification was deferred, but this is a hard blocker before processing real payment events.

## Middleware role enforcement vs. actual access control

The admin/seller gating logic has a fail-open condition:

```typescript
if (isAdminRoute(pathname)) {
  const userRole = request.cookies.get("user-role")?.value;
  if (userRole && userRole !== "admin" && userRole !== "super_admin") {
    return NextResponse.redirect(new URL("/dashboard", request.url));
  }
}
```

If the cookie is absent, the `if (userRole && ...)` check is skipped and the user passes through to the admin page. A user who deletes the `user-role` cookie while keeping their auth session cookie gains access to admin routes.

The comment says "Full role verification happens at the page level with server-side DB lookup" but all admin pages are `"use client"` components that render immediately with mock data and perform no server-side authorization.

## In-memory rate limiter in a serverless context

The bucket state lives in a module-level `Map`:

```typescript
const buckets = new Map<string, TokenBucket>();
```

On Vercel, each request may hit a different Lambda instance. The rate limiter will never accumulate enough hits to trigger a block. Additionally, the middleware adds `X-Client-IP` and `X-Request-Timestamp` headers but never actually calls `checkRateLimit` — the rate-limiting infrastructure is entirely disconnected from the request path.

## KYC verification disconnection

The server actions in `verification-actions.ts` authenticate the user, upload to Supabase storage, record the document, and update verification status. But the client component `verification-flow.tsx` never invokes them:

```typescript
const handleSubmit = async () => {
  setIsSubmitting(true);
  await new Promise((resolve) => setTimeout(resolve, 1500));
  setIsSubmitting(false);
};
```

The file upload inputs capture `File` objects into state but there's no `FormData` construction or server action call. The entire 4-step KYC wizard is visual only.

## Database design: escrow release concurrency

`process_escrow_release` checks `v_order.escrow_status != 'held'` and then updates the wallet balance and order status. Without a `SELECT ... FOR UPDATE` on the orders row, two concurrent calls (e.g., from a webhook retry and an admin manual release) could both see `escrow_status = 'held'`, pass the guard, and credit the seller wallet twice.

## RLS policy edge cases

The `device_sessions` table has a `FOR ALL USING (user_id = get_user_extended_id())` policy — a compromised session can delete its own session record to cover its tracks.

The `analytics_events` table allows `FOR INSERT WITH CHECK (true)` — any authenticated user can insert unlimited analytics events, a potential spam/storage-exhaustion vector without application-level rate limiting (which, per the above, doesn't function).

## Type safety gaps

The `database.types.ts` defines comprehensive Row/Insert/Update types for all 23 tables, but the service layer defines parallel type systems that don't reference them. When stubs are replaced with real Supabase queries, the types will need reconciliation.

The `ROLE_PERMISSIONS` object in `roles.ts` repeats permission arrays for each role rather than inheriting from the hierarchy. Adding a new permission requires updating every role above the one that should have it — an easy source of drift.
