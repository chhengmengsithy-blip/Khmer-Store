# Product creation flow with buyer-to-seller self-registration

Server actions for product insertion and seller registration, wired to React form components with a conditional server-component gate that checks seller status before showing the appropriate form. The auth callback route and sign-in actions were also enhanced to set a `user-role` cookie for middleware-based route gating.

Watch for:
- **Role escalation via client-controlled update** (confirmed) — `registerAsSeller` updates `users_extended.role` through RLS policies that allow any user to update their own row, meaning any authenticated user can set their role to "seller" without admin approval.
- **Stale role cookie after seller registration** (confirmed) — after `registerAsSeller` upgrades the role to "seller", the `user-role` cookie is never refreshed. Middleware gating will still see "buyer" until the next sign-in.
- **Slug collision possible in concurrent inserts** (likely) — the slug uniqueness relies on `Date.now().toString(36)` appended as a suffix, but there's no UNIQUE constraint on `products.slug` in the schema, so two products created in the same millisecond could collide silently.

## High-level view

The product creation flow adds two server actions — `createProduct` and `registerAsSeller` — that validate fields, resolve the seller record through a users_extended → sellers join, and insert into the respective tables. The RLS policies do the real authorization (INSERT on products requires that `seller_id` belongs to the current user), so the server actions serve as a convenience validation layer that gives friendly error messages before hitting the DB.

The page at `/dashboard/products/new` was converted to a server component that gates form rendering based on whether the authenticated user has a seller record. The auth callback route is new and handles Supabase email-confirmation code exchange plus role-cookie setting. However, `registerAsSeller` does not refresh this cookie, creating a window where the middleware's view of the user's role is stale.

The seller registration path has an authorization gap: since the RLS policy on `users_extended` allows users to UPDATE their own row without column restrictions, the action effectively lets any buyer self-promote to seller — or worse, to any role in the CHECK constraint.

1. **Self-serve role escalation** — `registerAsSeller` updates `users_extended.role` to "seller" and the RLS policy allows the user to update their own row with no column restriction. If seller approval is a business requirement, add a DB constraint or use a service-role client for this update.
2. **Stale user-role cookie after registration** — After upgrading to seller, the cookie still says "buyer" until the next login. Call `setUserRoleCookie` (or equivalent) after role upgrade in `registerAsSeller`, or use `cookies().set()` to push the new role.
3. **Slug uniqueness not enforced at DB level** — `products.slug` has no UNIQUE index. Add one (or a partial unique index on non-deleted rows) to prevent silent collisions.
4. **Category and condition not persisted** — The form collects `category` and `condition` in React state but never sends them to the server action or inserts them into the DB.
5. **No length/size bounds on text fields** — `title` and `description` have no max-length validation, so arbitrarily large payloads reach the DB. Add server-side length caps.

## Seller registration and the privilege escalation surface

The `registerAsSeller` action performs an INSERT into `sellers` (allowed by the RLS policy `"Users can create seller profile" WITH CHECK (user_id = get_user_extended_id())`) followed by an UPDATE to `users_extended` setting `role = 'seller'`. The RLS policy `"Users can update own extended profile" USING (auth_user_id = auth.uid())` permits this unconditionally — there is no column restriction preventing a user from setting their own role to any value within the CHECK constraint (which includes "admin", "super_admin", "moderator").

The application code only sets "seller", but the concern is structural: the same Supabase client that runs `registerAsSeller` could be called with a hand-crafted payload that sets an elevated role. If this marketplace requires admin-gated seller approval, the fix is either (a) a Postgres column-level grant that prevents the `authenticated` role from writing to `users_extended.role`, or (b) performing the role update with a service-role Supabase client that bypasses RLS, and removing the ability of the user's own session to touch the `role` column.

## Cookie lifecycle gap after role change

The `user-role` cookie is set at three points: sign-in, OTP verification, and the auth callback. It is never updated when `registerAsSeller` changes the role. After registration the user must sign out and back in for middleware to recognize them as a seller. If any middleware route requires `role === "seller"` to access seller-only pages, those pages will be inaccessible immediately after registration despite the success message.

The fix: after the role UPDATE succeeds in `registerAsSeller`, set the cookie:

```typescript
const cookieStore = await cookies();
cookieStore.set("user-role", "seller", ROLE_COOKIE_OPTIONS);
```

## Slug generation and collision window

`generateSlug` strips non-alphanumeric characters and appends `Date.now().toString(36)`. For Khmer-script product titles, the regex `[^a-z0-9\s-]` will strip the entire title, leaving a slug of just `-<timestamp>`. Two products created in the same millisecond with non-Latin titles would have identical slugs. The `products` table schema has no UNIQUE constraint on `slug`, so this would succeed silently and create ambiguous URLs.

Mitigations: add a UNIQUE index on `(slug) WHERE deleted_at IS NULL`, and consider using `crypto.randomUUID().slice(0, 8)` instead of (or in addition to) the timestamp for the suffix.

## Form fields collected but not persisted

The `SellerProductForm` component tracks `category` and `condition` in state and renders Select dropdowns for them, but `handleSubmit` never adds them to the FormData. The `products` table has a `category_id` FK and a `metadata` JSONB column where condition could be stored, but neither is wired up. The shipping checkboxes are similarly decorative — no state is tracked and nothing is submitted. Sellers will fill in category and condition and assume they're saved.

## Input validation bounds

`createProduct` does not enforce any maximum length on `title` or `description`. The DB schema defines both as `TEXT NOT NULL` with no length constraint, so a malicious client could send megabytes of text. Adding `title.length > 200` and `description.length > 10000` guards would prevent oversized payloads from reaching the DB.
