# V4 Professional Upgrade: Design system, marketplace, admin panel

A comprehensive UI overhaul across 57 files that transforms Khmer Store from a basic marketplace scaffold into a polished application. The change introduces a cohesive design token system in globals.css, creates 8 new shared components (DataTable, ImageUpload, Pagination, PriceInput, StatsCard, Breadcrumbs, LoadingSkeleton, ConfirmDialog), upgrades 4 existing components, and rewrites nearly every page — homepage through admin panel — with a dark-themed glassmorphism aesthetic. The architecture is sound: server components handle data fetching, client components manage local interaction state, and localStorage fills in where backend persistence hasn't been built yet.

**Watch for:** Client-side favorite state on ListingCard is fire-and-forget with no backend persistence (confirmed). Password change flow ignores the "current password" field — Supabase `updateUser` is called without verifying the existing credential (confirmed). Admin settings and user notification/privacy preferences are stored exclusively in localStorage, meaning they vanish on device change or incognito (confirmed). The DataTable sort state is tracked in UI but never applied to the underlying data (confirmed).

## High-level view

The design system introduces two parallel color schemes in globals.css — HSL custom properties for shadcn compatibility and raw hex CSS variables for the V4 branding. The dual-track approach works because the hex tokens are mapped into Tailwind's `@theme inline` block, but a future maintainer could easily break the mapping between `--color-surface` the root variable and `--color-surface` the theme binding since both have the same name.

All new shared components are well-typed with proper interfaces and sensible defaults. The DataTable is the most structurally complete — generic over its row type, supporting selection, sorting indicators, search, and pagination — but sort and search logic is purely cosmetic: the sort state is tracked without actually reordering data, and search fires a callback but doesn't filter locally.

The marketplace introduces URL-driven filtering via search params (category, condition, price range, sort, pagination). Server-side rendering fetches all listings then paginates client-side. The post listing form uses a 5-step wizard with localStorage draft persistence, properly excluding blob URLs from the draft.

The settings page exposes password change but does not verify the current password before calling `supabase.auth.updateUser()`. The `currentPassword` field is collected from the user but never sent to Supabase, meaning any authenticated session can change the password without re-authentication. Admin settings and notification/privacy toggles are localStorage-only — they have no server-side effect.

The admin sidebar gained collapsible behavior with localStorage persistence and grouped navigation sections. The admin dashboard, users, listings, and reports pages all use StatsCard and DataTable, giving the admin panel a consistent table-based management interface.

<summary>Issues (6)</summary>

1. **Favorite toggle has no persistence** — ListingCard toggles local state and calls an optional callback, but no component wires it to a backend mutation. Favorites will revert on page reload. Add server action integration or document this as intentional UI-only behavior.
2. **Password change skips current password verification** — `handleChangePassword` collects `currentPassword` but never sends it to Supabase. Anyone with an active session can reset the password. Either use Supabase's `signInWithPassword` first to re-authenticate, or remove the "current password" field to avoid false confidence.
3. **DataTable sort is cosmetic** — `handleSort` updates sort key/direction state but the `data` array passed to the table is rendered in its original order. Either sort the data before rendering or document that consumers must handle sorting externally.
4. **localStorage-only preferences** — Notification preferences, privacy toggles, and admin site settings are persisted exclusively in localStorage. They have zero server-side enforcement (e.g., "show phone publicly" doesn't influence what the API returns). Clarify in UI that these are device-local preferences or back them with a database column.
5. **ImageUpload blob URLs leak on remove** — When an image is removed, `URL.revokeObjectURL` is never called on the discarded blob URL, leaking browser memory. Call `revokeObjectURL` in `handleRemove`.
6. **Sitemap uses static `new Date()` at build time** — Every route's `lastModified` is set to `new Date()`, which means every build produces a sitemap claiming all pages changed "now". This is harmless but misleading to crawlers. Consider using a real last-modified date or a static date for truly static pages.

## Favorite toggle on ListingCard: local-only state

The ListingCard was upgraded from a server component to a client component to support image loading skeleton and a heart-toggle favorite button. The `favorited` state initializes from the `isFavorited` prop but toggles purely in local state:

```tsx
const [favorited, setFavorited] = useState(isFavorited);

const handleFavorite = (e: React.MouseEvent) => {
  e.preventDefault();
  e.stopPropagation();
  setFavorited(!favorited);
  onFavoriteToggle?.(listing.id); // callback, but no caller wires it to a server action
};
```

No consumer of `ListingCard` on the homepage, marketplace, or listing detail pages passes `onFavoriteToggle`. The visual toggle is satisfying but misleading — the heart reverts on any re-render or navigation. This is a UX gap: users will believe they've saved an item when nothing was persisted. Confidence: **confirmed** — traced through all callsites.

## Password change ignores the current password field

In `src/app/dashboard/settings/page.tsx`, the form collects `currentPassword` but the submission handler only calls:

```tsx
const { error } = await supabase.auth.updateUser({
  password: newPassword,
});
```

Supabase's `updateUser` operates on the current session token and does not require re-authentication. The current-password field provides a false sense of security — it's never validated. An attacker who inherits an active session (e.g., shared computer, stolen token) can change the password without knowing the original. Confidence: **confirmed**.

## DataTable sort and search contract

The DataTable component is well-structured as a generic `<T>` component, but its sort behavior is half-implemented. Clicking a sortable column header updates `sortKey` and `sortDirection` state, and the UI renders up/down chevrons accordingly. However, the `data` prop is iterated directly in render order:

```tsx
data.map((item) => { ... })
```

No sorting function is applied. The component's API suggests sorting happens externally (via `onSearch` callback for search), but there's no `onSort` callback. A consumer would need to observe the sort UI visually and have no way to react to it. Either the table should sort its own data by `item[sortKey]` (with a type constraint), or it should expose an `onSortChange` callback. Confidence: **confirmed**.

## ImageUpload blob URL memory leak

When images are removed via `handleRemove`, the filtered array replaces the image list, but the blob URL created by `URL.createObjectURL` is never revoked:

```tsx
const handleRemove = (index: number) => {
  const newImages = images.filter((_, i) => i !== index);
  onChange(newImages);
  // Missing: URL.revokeObjectURL(images[index])
};
```

For a form where a user might add and remove 10+ images while composing a listing, this leaks memory. The same issue exists in the reorder functions (`handleMoveLeft`/`handleMoveRight`) — those don't create new URLs, so they're fine. Confidence: **confirmed**.

## Client-side pagination on marketplace

The marketplace page fetches listings server-side with `getListings()` but slices the array client-side for pagination:

```tsx
const ITEMS_PER_PAGE = 12;
const startIndex = (page - 1) * ITEMS_PER_PAGE;
const paginatedListings = filteredListings.slice(startIndex, startIndex + ITEMS_PER_PAGE);
```

This means all listings matching the query are fetched on every page view, even though only 12 are displayed. For now this is acceptable since `getListings` returns an empty array when Supabase isn't configured, and the dataset is likely small in early stages. But this won't scale past ~500 listings without server-side pagination. The architecture makes this easy to fix later — `getListings` already accepts a `limit` param. Confidence: **confirmed** (read the marketplace page.tsx and listing-actions pattern).

## Admin settings: localStorage as "database"

The admin settings page (`/admin/settings`) stores site configuration — site name, description, commission rate, maintenance mode — in localStorage:

```tsx
function handleSave() {
  localStorage.setItem("admin-site-settings", JSON.stringify(settings));
}
```

The maintenance mode toggle has no server-side enforcement; toggling it produces a visual badge but doesn't block any routes. The user notification and privacy preferences (`notification_prefs`, `privacy_prefs`) have the same issue — "show phone publicly" is a UI toggle that doesn't influence the API's response when another user views the listing. This is acknowledged in the task findings as "UI-only toggles stored in localStorage," so it's intentional for now, but the settings pages present these as real configurations without a "coming soon" or "local only" disclaimer. Confidence: **confirmed**.

## Newsletter and contact forms: no backend

The homepage newsletter subscription input and contact page form collect user input but have no submission handler — the buttons are decorative. The newsletter lacks even an `onSubmit` handler; the subscribe button does nothing on click. The contact form has client-side validation and a simulated submission delay via `setTimeout`, showing a success state without transmitting data. These are fine as UI scaffolds but should either be wired to an API or clearly marked as non-functional. Confidence: **confirmed**.
