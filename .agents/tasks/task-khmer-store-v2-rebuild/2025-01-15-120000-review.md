# Classified ads marketplace rebuild (V2)

The entire Khmer Store frontend has been gutted and rebuilt: from a "luxury marketplace" with 3D scenes, mock data, admin panels, checkout flows, and role-based access into a classified ads platform (Khmer24-style) with categories, empty-state-driven pages, a post-listing form, and graceful degradation when Supabase is not configured. The removal is ~8200 lines; the addition is ~2400 lines. What remains is a navigable shell: every page renders, every button does something, but no real data flows through yet.

**Watch for:** Sign-up form never actually calls `signUp` — it shows a success toast without persisting anything (confirmed). The `isSupabaseConfigured` check in middleware lets unauthenticated users through all "protected" routes in preview mode, meaning the `/post` form and `/dashboard` are open to everyone when env vars are absent (confirmed). Homepage featured-listings section has an identical branch for both `supabaseReady` true/false (confirmed, dead code).

## High-level view

The 3D dependencies and every fake-data component have been deleted. In their place is a 9-category taxonomy defined in `constants/categories.ts` that drives the homepage grid, the header dropdown, the category detail pages, the filter sidebar, and the post-listing form — a single source of truth for the site's information architecture.

Authentication now has two modes: when Supabase env vars are present it works normally; when they are absent, `createClient()` returns `null` and every consumer either shows a warning banner or silently skips auth. The middleware's protected-route guard is bypassed entirely in this state, making "preview mode" effectively anonymous-access mode.

The sign-up form validates inputs client-side but never calls the `signUp` server action even when Supabase is configured. It fires a toast pretending the account was created. This makes the sign-up flow non-functional regardless of configuration.

The `ListingCard` component exists with full formatting logic (time-ago, price formatting, condition labels) but is never instantiated anywhere — an unused but forward-looking shell.

The `isSupabaseConfigured()` helper is duplicated across client and middleware modules. Both check the same `NEXT_PUBLIC_*` vars, but the duplication creates a maintenance risk if the check ever diverges between Edge and browser runtimes.

## Sign-up form doesn't call the server action

The `SignUpForm` component imports `useToast` but does not import `signUp` from the auth actions. When `supabaseReady` is true and validation passes, it executes:

```tsx
try {
  toast({
    title: "Account created",
    description: "Check your email to verify your account.",
  });
} catch {
  setError("An unexpected error occurred. Please try again.");
}
```

No network call. No server action. The user sees "Account created" but nothing was persisted. This is **confirmed** — the import of `signUp` was removed during the rewrite and never re-added. The sign-in form still calls `signIn` correctly, so this asymmetry between the two auth forms is almost certainly an oversight rather than a deliberate stub.

## Middleware preview-mode bypass

`src/middleware.ts` now has:

```ts
if (isProtectedRoute(pathname)) {
  if (!isSupabaseConfigured()) {
    return response; // <-- full bypass
  }
  // ...auth cookie check...
}
```

The protected prefixes are `/dashboard`, `/messages`, and `/post`. In preview mode (no env vars), any anonymous visitor can access these pages without friction. The post form shows the amber warning and refuses to submit, so the data-loss risk is low — but the dashboard settings page renders form fields for profile editing, which could confuse users into thinking their changes will save. **Confirmed** by reading the middleware diff.

## Dead-code branch on the homepage

```tsx
{!supabaseReady ? (
  <EmptyState icon={Package} title="No listings yet" ... />
) : (
  <EmptyState icon={Package} title="No listings yet" ... />
)}
```

Both arms are identical — props, copy, everything. This is a placeholder for "when Supabase is ready, fetch and render real listings here." Collapse to one unconditional render or differentiate the `else` arm with a data-fetching component. **Confirmed** at `src/app/page.tsx` lines 117-131.

## Object URL leak in photo upload

`PostListingPage.handlePhotoChange` creates blob URLs for previews:

```tsx
newPhotos.push(URL.createObjectURL(files[i]));
```

These are never revoked. If a user selects photos repeatedly or navigates away, the blobs stay in memory. The component has no cleanup effect. **Likely** leak in extended usage patterns — add `URL.revokeObjectURL` on unmount or when photos are replaced.

## Duplicated `isSupabaseConfigured` across runtimes

The function is defined identically in `src/lib/supabase/client.ts` (browser) and `src/lib/supabase/middleware.ts` (Edge). The server module (`server.ts`) imports from `./client`. If the check ever needs to reference a non-`NEXT_PUBLIC` secret, the middleware copy would need independent updating. **Possible** maintenance concern — consider extracting to a shared `supabase/config.ts` that both import.
