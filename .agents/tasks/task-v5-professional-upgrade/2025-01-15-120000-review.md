# V5 professional upgrade: dashboard sub-pages, SEO metadata, and documentation

This change splits the dashboard into proper sub-routes (`/dashboard/listings` and `/dashboard/favorites`), adds `<Metadata>` exports to several public pages, expands the sitemap and robots.txt coverage, and rewrites the README for V5. The approach is straightforward: new client components for the two dashboard pages, layout files to inject metadata on client-component routes (contact, post), and sitemap/robots refinements. The site config's `ogImage` now points to `/logo.svg`.

Watch for: silent failure on data fetch in both dashboard pages (confirmed), the `ogImage` change to a relative path that won't resolve correctly in social sharing contexts (confirmed), and delete operations that swallow errors without user feedback (confirmed).

## High-level view

The two new dashboard pages fetch data inside `useEffect` without `.catch()` handlers, so any rejected promise (network error, auth expiry, Supabase outage) will leave the page stuck on "Loading..." indefinitely with no error indicator. The favorites page additionally casts join results through an unsafe `Record<string, unknown>` intermediate that will silently drop malformed rows rather than surfacing a data integrity problem.

The `ogImage` in `src/config/site.ts` was changed from an absolute URL (`https://khmerstore.com/og.png`) to a relative path (`/logo.svg`). Open Graph and Twitter Card validators require a fully-qualified URL; a relative path will fail to render a preview on most social platforms.

The delete flow in the listings page removes a listing from local state on success but shows no toast or feedback on failure. The user clicks "Delete", sees the confirm dialog close, and the listing remains in the list with no explanation.

The dashboard sub-pages are client components under a client-component layout, so they cannot export Next.js metadata — unlike the contact and post routes which got layout wrappers for exactly this purpose. These pages will show the root-level title in the browser tab.

1. **Unhandled promise rejection on dashboard pages** — Both `/dashboard/listings` and `/dashboard/favorites` call server actions inside `useEffect` without a `.catch()`. A network or auth error leaves the page in a permanent loading state. Add `.catch()` and set an error state.

4. **Favorites data casting is fragile** — The favorites page maps join results through `f.listings as Listing | null` after typing `f` as `Record<string, unknown>`. If the Supabase join shape changes (e.g., relation renamed), this silently returns an empty list rather than surfacing a type error.

## Missing error boundaries on data-fetching effects

Both dashboard pages call a server action inside `useEffect` without attaching a `.catch()`:

```tsx
getUserListings().then((data) => {
  if (mounted) {
    setListings(data);
    setLoading(false);
  }
});
```

If `getUserListings` throws (network failure, Supabase client misconfiguration), the promise rejects unhandled, `setLoading(false)` never fires, and the user sees "Loading your listings..." permanently. The same pattern appears in `dashboard/favorites/page.tsx`. An `error` state with a retry button would give users a recovery path. [confirmed]

## ogImage relative path breaks social previews

`src/config/site.ts` now sets `ogImage: "/logo.svg"`. OpenGraph requires a fully-qualified URL per the protocol spec. Facebook's crawler, Twitter's card validator, and LinkedIn's post inspector will all fail to resolve a relative path. The fix is to use `` `${siteConfig.url}/logo.svg` `` or assign it after `url` is defined. Additionally, SVG is not widely supported as an OG image format — most platforms expect PNG or JPEG. [confirmed]

## Delete operation feedback gap

In the listings page, `handleDelete` removes a listing from local state on success but does nothing on failure:

```tsx
const result = await deleteListing(deleteId);
if (!result.error) {
  setListings((prev) => prev.filter((l) => l.id !== deleteId));
}
setDeleting(false);
setDeleteId(null);
```

When `result.error` is truthy, the dialog closes and the listing remains in the list with no explanation. This should surface the error — a toast notification or keeping the dialog open with an error message. [confirmed]

## Sitemap includes auth routes

The sitemap now includes `/sign-in`, `/sign-up`, and `/forgot-password`. Google's guidance recommends against indexing pages that serve no content to unauthenticated crawlers. If these pages render meaningful SEO content (marketing copy, trust signals), this is fine; if they're bare login forms, excluding them would be more appropriate. [possible — check whether these pages render meaningful content for crawlers]

## Dashboard layout metadata gap

Because the dashboard layout is a `"use client"` component, it cannot export `metadata`. The new sub-pages are also client components. For other client-component routes (contact, post), the PR correctly uses a parent `layout.tsx` with metadata. The dashboard pages lack this treatment — their browser tab will show the root-level title rather than "My Listings | Khmer Store". [confirmed]
